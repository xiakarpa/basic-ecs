# basic-ecs

Базовая реализация ECS-системы с интегрированным скриптингом на lua. 

## Сборка и запуск

Проект собирается через CMake, необходимые зависимости автоматически подгружаются через FetchContent

Проект собирался под Ubuntu 24.04 (WSL)

* Компилятор: g++-13
* Стандарт: C++17
* Минимальная версия CMake: 3.28

### Сборка
Скачайте репозиторий с исходным кодом

В корневой директории репозитория введите команды:
```sh
cmake -B <path>
cmake --build <path>
```

### Запуск
В корневой директории репозитория введите команды:
```sh
./<path>/bin/basic-ecs
```

## Комментарии  

### Указатели
Для компонент и других сущностей был выбран shared_ptr. 

Для однообразия я использовал везде только умные указатели, так как смешивание указателей это не очень хороший тон, особенно по части стилизации кода.

Почему указатели на сцену и провайдера являются shared понять не трудно - это должна быть одна сущность со множеством референсов.

**Почему shared_ptr для компонентов**

>В самом начале для меня показалось очевидным хранить компоненты как unique_ptr, так как ими должен владеть только менеджер. Но у этого подхода возникли проблемы, которых мне захотелось избежать. 
\
\
Во первых, может возникнуть ситуация, когда два разных Entity используют один и тот же компонент - например, если это позиция игрока, которую использует сущность Player и сущность Enemy для искусственного интеллекта. Пришлось бы делать мосты для синхронизации данных между ними, либо делать отдельные компоненты. А shared_ptr позволяет отслеживать данные из коробки.
\
\
Во вторых, мы отдаем указатели через провайдер, которые должны быть мутабельными (иметь возможность меняться). Либо нужно делать систему сеттеров (вероятно на огромных шаблонах), либо отдавать weak_ptr, что в целом нормально, но сразу вызывает проблемы с проверкой жив ли этот указатель и не позволяет компонентам перейти в другое владение, например, если мы захотим сделать еще один менеджер компонентов для какой-нибудь специфической физики, но чтобы они при этом были и в основном менеджере для отрисовки.
\
\
По итогу shared_ptr позволяет гораздо проще работать с отслеживанием данных и их изменением извне, на что я полагался при написании систем.

### Интеграция lua
Для интеграции я использовал библиотеку sol2. Я решил немного расширить функционал, о котором просили, и сделал скрипт для любого Entity, вместо конкретного, чтобы это можно было использовать в воображаемом game loop'е.

Сам биндинг состоит из функций, которые прокидываются в lua; инициализации, которая связывает эти функции в c++ и в lua; и функций, которые уже можно использовать в main-файле.

Это приводит к небольшому бойлерплейту в виде той самой инициализации, но учитывая специфику C++ - это в любом случе придется делать, поэтому я посчитал, что я лучше явно разобью эти вещи и предложу более-менее удобную структуру. Как раз в том числе поэтому весь биндинг lua находится в одном файле - чтобы можно было сразу полностью описать связь нового скрипта не копаясь по разным файлам.

### Шаблоны (если очень интересно почитать мои рассуждения)
При реализации ECS часть функций по типу getComponent реализована на шаблонах, что может выбиваться из остальной структуры, однако это очевидное решение проблемы поиска необходимых данных - в случае использования шаблонов и typeid нам не нужно заботиться ни о максимальном количестве компонентов, ни об обозначении их типов. Единственная пробема: невозможность присвоить одному entity несколько одинаковых компонентов, однако это решается простым наследованием компонента, присваивая ему другое имя. Можно было бы сделать систему имён для компонентов, но это бы лишний раз использовало неконстантные строки, количество которых я хотел сократить. 

Плюс из идеи шаблонов появился концепт наследования System<Component1, Component2, ...> как гарантии наличия этих компонентов у сущностей, которые может получить система

Я в целом много раз переписывал именно эту связь между System и Component, вращая зависимости между ними в разные стороны и по итогу, как мне кажется, нашел оптимальный вариант в виде интерфейсов+отдельного провайдера, который при этом соответствовал требованиям ТЗ.
